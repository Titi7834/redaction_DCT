# ADR-002 — Adopter une idempotence par clé client (`Idempotency-Key`) pour les opérations d'écriture sensibles

**Date** : 2026-04-29
**Auteurs** : Tom LEPRIEUR, Arthur L'AFFETER, Tiago DA COSTA

## Contexte

Plusieurs opérations critiques de SupEvents — réservation d'un billet, création d'un PaymentIntent, traitement d'un webhook Stripe — peuvent légitimement être rejouées par un client (double-clic utilisateur, retry réseau côté SDK, retry policy interne d'un consommateur) sans que cela traduise une intention nouvelle de l'utilisateur. Sans contrôle d'idempotence, ces rejeux produisent des doublons : deux billets pour une même intention de réservation, deux paiements créés pour un même billet, deux exécutions du même handler de webhook.

Forces en présence :

- **Sécurité métier** : aucune opération financière ou allocation de ressource ne doit être exécutée plus d'une fois par intention utilisateur.
- **Tolérance aux erreurs réseau** : un client qui ne reçoit pas la réponse d'une requête doit pouvoir la rejouer sans risque.
- **Conformité aux standards** : Stripe, Square, AWS imposent ou recommandent un mécanisme d'idempotence par en-tête `Idempotency-Key` ; aligner SupEvents sur ce standard facilite l'intégration.
- **Coût d'implémentation** : un mécanisme d'idempotence universel transparent (sans contrat avec le client) reste théorique — toute approche pragmatique exige une convention.

## Décision

Les endpoints d'écriture sensibles (`POST /events/{eventId}/tickets`, `POST /tickets/{ticketId}/payment-intent`) acceptent un en-tête HTTP **`Idempotency-Key`** porté par le client (UUID v4 généré côté front avant émission de la requête). Le serveur stocke en Redis le couple `(key, hash(payload))` → `réponse complète`. Toute requête ultérieure portant la même clé renvoie la réponse cachée si le hash du payload est identique, ou un `422 Unprocessable Entity` si le payload diffère (réutilisation abusive). Pour les webhooks Stripe entrants, l'idempotence est portée par `event.id` Stripe (déduplication serveur native, pas de clé client).

## Alternatives envisagées

### Déduplication serveur seule (sans en-tête client)

Le serveur calcule un hash du `(userId, eventId, timestamp arrondi)` ou similaire pour détecter les rejeux, sans coopération du client.
*Avantages* : aucun changement côté front, transparent.
*Raison de l'écart* : aucune heuristique côté serveur ne distingue de façon fiable un rejeu (« le même utilisateur réserve deux fois en 100 ms, est-ce volontaire? ») d'une vraie double-réservation. Trop laxiste → on bloque des cas légitimes ; trop strict → on laisse passer des doublons. La clé client est la seule source d'information fiable sur l'intention.

### Verrou applicatif sur l'utilisateur ou la ressource

`SET NX EX user:{id}:lock=1 ttl=10s` autour de la requête.
*Avantages* : simple à implémenter, pas de changement de contrat API.
*Raison de l'écart* : un verrou n'est pas une idempotence — il bloque les requêtes pendant sa durée de vie mais ne renvoie pas le résultat de la première au lieu de la deuxième. Si le client n'a pas reçu la réponse de la première à cause d'un timeout réseau, le verrou la bloque sans lui rendre le résultat. Solution boiteuse pour le cas d'usage réel.

### Aucun mécanisme dédié, double-clic neutralisé côté front uniquement

S'appuyer sur le bouton désactivé après clic + la contrainte `UNIQUE (user_id, event_id)` de PostgreSQL pour rejeter les doublons en base.
*Avantages* : zéro coût d'infra.
*Raison de l'écart* : ne couvre pas les retries SDK Stripe, ne couvre pas un utilisateur qui rafraîchit une page de tunnel de paiement, ne couvre pas les retries de consommateurs RabbitMQ. La contrainte UNIQUE produit un 409 mais ne renvoie pas l'objet créé par la première requête, ce qui dégrade l'UX.

## Conséquences

### Positives

- Alignement avec le standard de fait du secteur (Stripe, AWS, GitHub) — courbe d'apprentissage des intégrations tierces facilitée.
- Modèle uniforme côté API : un seul mécanisme à comprendre pour tous les endpoints sensibles.
- La clé client porte l'intention utilisateur de façon explicite et auditable.
- Résiste aux double-clics, aux retries réseau et aux retries broker simultanément.

### Négatives

- Contrainte sur le client : front + clients tiers doivent générer et propager la clé. Documentation API à soigner.
- Stockage Redis additionnel : `(key, hash, response)` × nombre de requêtes idempotentes × TTL. Volumétrie maîtrisable mais à monitorer (TTL hypothèse V1 quelques heures).
- Cas d'erreur supplémentaire à exposer : `422 IDEMPOTENCY_REPLAY` quand la clé est réutilisée avec un payload différent. Comportement à expliquer dans la documentation API.
- Le hash du payload doit être calculé sur une représentation canonique stable (ordre des clés JSON, normalisation des espaces) — sinon faux positifs sur réutilisation abusive.

### Dette acceptée

Le mécanisme repose sur Redis : si Redis est indisponible au moment d'une requête idempotente, deux choix possibles — soit fail-open (laisser passer la requête sans contrôle, risque de doublon), soit fail-closed (rejeter la requête, indisponibilité fonctionnelle). En V1, retenu **fail-closed** avec circuit breaker court (la requête échoue avec un 503 explicite plutôt qu'accepter un doublon silencieux). Un mécanisme de fallback sur PostgreSQL (table `idempotency_keys` répliquée) est explicitement reporté à une itération ultérieure si la disponibilité Redis devient un point de douleur.
