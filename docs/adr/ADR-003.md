# ADR-003 — Adopter une authentification JWT stateless courte + refresh token rotatif plutôt que des sessions serveur via Redis

**Date** : 2026-04-29
**Auteurs** : Tom LEPRIEUR, Arthur L'AFFETER, Tiago DA COSTA

## Contexte

SupEvents s'authentifie via le SSO école (OpenID Connect). Une fois l'`id_token` IdP validé, le backend doit émettre un mécanisme d'authentification applicatif que le front présentera à chaque requête. Deux familles de solutions s'affrontent dans l'industrie :

- **JWT stateless** : un access token signé, court, autoportant ; pas de lookup serveur à chaque requête, mais révocation difficile.
- **Sessions serveur** : un identifiant opaque envoyé au client, l'état de session étant stocké côté serveur (typiquement Redis) ; révocation triviale, mais lookup à chaque requête.

Forces en présence dans le contexte SupEvents :

- **Architecture multi-services** : plusieurs modules (`AuthModule`, `EventModule`, `TicketModule`...) doivent valider l'authentification. Un mécanisme stateless évite que chaque module dépende du store de sessions.
- **Latence** : un lookup Redis à chaque requête est faible (sub-ms LAN) mais non nul, et constitue un point de dépendance critique.
- **Sécurité de la révocation** : un compte compromis ou révoqué doit pouvoir être bloqué immédiatement, sans attendre l'expiration naturelle du jeton.
- **Scalabilité horizontale** : plusieurs instances backend derrière un load balancer doivent toutes pouvoir valider un jeton sans état partagé.

## Décision

L'authentification applicative repose sur un mécanisme **JWT stateless court + refresh token rotatif** :

- **Access token JWT** signé (algorithme RS256 ou EdDSA — choix précis à figer en § 9), courte durée de vie (**hypothèse V1 : 15 minutes**), contenant les claims `sub` (`User.id`), `role`, `exp`, `iat`, `jti`.
- **Refresh token opaque** émis simultanément, longue durée de vie (**hypothèse V1 : 7 jours**), stocké en cookie `HttpOnly + Secure + SameSite=Strict`, hashé en base (`RefreshToken.token_hash`, cf. § 6.4), rotaté à chaque usage.
- **Révocation** : pour les cas critiques (logout, compte compromis, admin revoke), un denylist court de `jti` est stocké en Redis avec TTL = durée de vie restante du JWT — le coût Redis ne porte que sur les jetons révoqués activement, pas sur tous.

## Alternatives envisagées

### Sessions serveur opaques en Redis

Le client reçoit un `session_id` opaque, le serveur stocke `(session_id → user_id, role, expires_at)` en Redis. À chaque requête, lookup Redis.
*Avantages* : révocation immédiate triviale (`DEL session:{id}`), pas de risque de fuite de claims dans un JWT décodable, possibilité de modifier les claims sans régénérer le jeton.
*Raison de l'écart* : Redis devient une dépendance critique sur le chemin de **toute** requête authentifiée — une indisponibilité Redis = indisponibilité totale de l'API. Cohérence cache/source-of-truth à maintenir entre PostgreSQL (`User.role`) et Redis. Ne se prête pas naturellement à un futur découpage en microservices où chaque service voudrait valider l'authentification sans appeler `AuthService`.

### JWT stateless sans denylist (révocation par expiration uniquement)

Variante stricte du stateless : on accepte que la révocation prenne effet à l'expiration naturelle du jeton.
*Avantages* : simplicité maximale, aucune dépendance Redis pour l'authentification.
*Raison de l'écart* : un délai de plusieurs minutes entre une révocation administrative (ex. compte compromis) et son effet réel est inacceptable côté sécurité. La durée du JWT devrait être réduite à quelques secondes pour compenser, ce qui annule l'avantage de performance.

### Sessions serveur en PostgreSQL

Variante de l'alternative 1 mais sans Redis, en utilisant une table `session` PostgreSQL.
*Avantages* : une seule technologie, cohérence transactionnelle.
*Raison de l'écart* : charge PostgreSQL multipliée par le nombre de requêtes authentifiées. Latence supérieure à Redis. Complexité opérationnelle équivalente sans le bénéfice de la stateless.

## Conséquences

### Positives

- Lookup Redis évité sur le chemin nominal (97 % des requêtes ne sont pas concernées par la denylist).
- Validation distribuée : chaque module backend peut valider un JWT avec la seule clé publique IdP (cohérent avec un futur découpage en microservices).
- Refresh token rotatif + détection de réutilisation (cf. § 7.1) protège contre le vol de jeton longue durée.
- Cookie `HttpOnly` empêche l'exfiltration du refresh token par XSS.

### Négatives

- Révocation non instantanée pour les access tokens non listés en denylist : un access token reste valide jusqu'à son expiration naturelle (max 15 min en V1). Acceptable car la durée est courte ; inacceptable serait une durée d'1 h.
- Taille des en-têtes HTTP : un JWT signé fait typiquement 700-1000 octets, transmis à chaque requête. Surcoût bande passante ~1 Ko par appel API ; négligeable sur Wi-Fi/4G mais à surveiller en mobile dégradé.
- Gestion des claims : un changement de rôle utilisateur (`user → organizer`) ne prend effet qu'au prochain refresh — délai de propagation maximal = durée de vie du JWT.
- Complexité côté front : gestion de la rotation du refresh token, gestion de la course critique entre deux requêtes parallèles qui détectent toutes deux un access token expiré.

### Dette acceptée

Le risque de **réutilisation simultanée légitime du même refresh token** (deux onglets ouverts, deux requêtes parallèles arrivant après expiration de l'access) est mitigé par une fenêtre de grâce courte côté serveur (le refresh token précédent reste accepté pendant quelques secondes après rotation), mais pas totalement éliminé. Si le problème devient récurrent en exploitation, deux pistes possibles : (1) verrou applicatif côté front sur la requête de refresh, (2) abandon du « one-time use » du refresh token au profit d'une fenêtre de réutilisation. La décision sera prise sur métriques.
